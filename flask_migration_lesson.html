Lesson: Flask Database Migrations with Flask-Migrate

In this lesson, we move from simply defining SQLAlchemy models to managing database schema changes in a professional Flask application. You will review ORM basics, understand how a Flask project is typically organized, see how blueprints fit into that structure, and then walk through the migration workflow from the very first migration to later schema updates.

1. ORM Refresher

In earlier lessons, we used SQLAlchemy as an ORM, which stands for Object-Relational Mapper. An ORM lets us represent database tables as Python classes and rows as Python objects. Instead of writing raw SQL for every insert, update, or query, we work with model classes such as Student, Assignment, and Grade. Each model class describes the structure of a table through columns, relationships, and constraints. This approach makes the code easier to read, easier to organize, and more consistent with the rest of the Python application. However, once the database already exists, changing a model class does not automatically update the existing table structure, and that is exactly the problem migrations solve.

Key idea: ORM models describe the schema we want in Python, but migrations are what keep the real database aligned with those model changes over time.

2. Typical Flask Application Structure

As Flask applications grow, it becomes important to separate responsibilities into different files and folders. A common structure includes an application factory, a configuration file, an extensions file, models, routes, and sometimes multiple blueprints. This structure keeps the app modular and easier to understand. 

flask-orm/
├── src/
│   └── migration_demo/
│       ├── __init__.py
│       ├── app.py
│       ├── config.py
│       ├── extensions.py
│       ├── models.py
│       ├── routes.py
│       ├── manage.py
│       ├── migrations/ <- created by Flask-Migrate
│
└── data/

What each file usually does

  • app.py: creates and configures the Flask app
  • config.py: stores settings such as the database URI
  • extensions.py: creates reusable objects like db and migrate
  • models.py: defines SQLAlchemy model classes
  • routes.py: contains view functions and API endpoints
  • manage.py: gives Flask CLI access to the app for commands such as migrations and run

Why this structure helps

  • It separates configuration from business logic.
  • It keeps model code away from route code.
  • It makes testing easier.
  • It scales better when the project grows.
  • It works well with extensions like Flask-Migrate.

3. What Is a Blueprint?

A blueprint is a way to organize related routes into reusable components. Instead of putting every route in one massive file, we can group related routes together and register them with the application. For example, one blueprint might handle students, another assignments, and another authentication. Blueprints help keep route code cleaner and make it easier to split a project into logical modules. They do not replace the Flask app itself, but they plug into it. 

from flask import Blueprint

api = Blueprint("main", __name__)

@api.get("/students")
def get_students():
    return {"message": "students endpoint"}

Then the blueprint is registered in the application factory:

from flask import Flask
from .routes import api

app = Flask(__name__)
app.register_blueprint(api)
Why it matters here: migrations work at the application and model level, but the routes that expose your data to users are often organized inside blueprints.

4. What Is a Migration?

A migration is a tracked, repeatable change to the database schema. Instead of deleting the database and starting over every time a model changes, we generate migration scripts that describe exactly how the schema should evolve. In Flask projects, Flask-Migrate provides the Flask integration, while Alembic generates and applies the migration scripts.

Why not just use db.create_all()? Because it is fine for quick demos, but it does not keep a history of changes, does not document the evolution of the schema, and does not safely handle ongoing updates in a real project.
  • Migrations track schema history.
  • Migrations produce scripts that can be reviewed.
  • Migrations can be applied consistently across environments.
  • Migrations help teams evolve a schema without guessing what changed.

5. Migration Workflow with Examples

The migration workflow usually follows the same pattern: define models, initialize migration support, generate a migration, apply it, and repeat when the schema changes. The examples below walk through the typical lifecycle.

Example 1: What migration files are

Migration files are Python scripts generated by Alembic. Each file records one schema change, such as creating tables, adding a column, dropping a constraint, or changing a field definition. These files become part of your project history and should usually be committed to version control. 

migrations/
├── env.py
├── script.py.mako
└── versions/
    ├── 2f1c9a_initial_schema.py
    └── 8e4d7b_add_due_date_to_assignment.py

The versions/ folder is especially important because it contains the actual migration scripts that describe how the schema changed over time.

Example 2: Initialize migration support

Before generating any migration files, the app must connect Flask-Migrate to the Flask app and the SQLAlchemy database object. 

from flask_sqlalchemy import SQLAlchemy
from flask_migrate import Migrate

db = SQLAlchemy()
migrate = Migrate()
def create_app():
    app = Flask(__name__)
    app.config.from_object(Config)

Then initialize the migration environment once: 

flask --app manage.py db init

This creates the top-level migrations/ directory that Alembic will use.

Example 3: Create the initial migration

Suppose the app already has these models: 

class Student(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(100), nullable=False)
    email = db.Column(db.String(120), unique=True, nullable=False)

class Assignment(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(120), nullable=False)
    max_score = db.Column(db.Integer, nullable=False)

Once the models exist, generate the first migration: 

flask --app manage.py db migrate -m "initial schema"

Flask-Migrate inspects the models and creates a migration file that contains operations such as create_table().

Example 4: Apply the initial migration

Generating a migration file does not yet change the database. To apply the migration to the actual database, run: 

flask --app manage.py db upgrade

This executes the migration script and creates the database tables described in the initial migration. At this point, the database and the model definitions are finally synchronized.

Important: db migrate creates the script. db upgrade actually applies it.
Example 5: Track schema changes later

Now imagine the requirements change and we add a due date to assignments. 

class Assignment(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(120), nullable=False)
    max_score = db.Column(db.Integer, nullable=False)
    due_date = db.Column(db.Date, nullable=True)

After changing the model, generate a new migration: 

flask --app manage.py db migrate -m "add due_date to assignment"

Then apply it: 

flask --app manage.py db upgrade

Alembic now adds only the new column instead of rebuilding the entire database. That is what makes migrations so powerful: they track schema evolution step by step.

6. Typical Commands

flask --app manage.py db init
flask --app manage.py db migrate -m "initial schema"
flask --app manage.py db upgrade

Later, after changing a model:

flask --app manage.py db migrate -m "describe the schema change"
flask --app manage.py db upgrade
Common classroom reminder: changing a model file is not enough. Students must generate a migration and apply it, or the database schema will still reflect the old version.

7. Putting It All Together

What the model does

Defines the schema in Python using classes and columns.

What the migration file does

Records the exact schema operations needed to update the real database.

What db migrate does

Generates a migration script by comparing current models to the existing schema state.

What db upgrade does

Runs the migration script against the database so the schema actually changes.

In short, SQLAlchemy gives us a Python-based way to describe our data model, while Flask-Migrate and Alembic give us a disciplined way to evolve that model over time. In a professional project, migrations are the bridge between model code and the real database.
